# Деплой Lexis

Кратко: как поднять **Postgres + Go API + статический фронт (nginx)** через Docker Compose из корня репозитория.

---

## Что нужно на машине

- **Docker** и **Docker Compose** v2 (`docker compose`).
- В продакшене — домен и возможность слушать порты **80/443** (или свой фронт-порт за reverse proxy).

---

## Быстрый старт (полный стек)

Из **корня** репозитория:

```bash
cp .env.example .env
# Отредактируй .env: хотя бы JWT_SECRET и пароль БД для продакшена.

docker compose up --build -d
```

Ожидаемое:

| Сервис    | Описание                         |
|-----------|----------------------------------|
| `db`      | Postgres 16, том `lexis-db-data` |
| `backend` | API на `:8080` **внутри** сети compose |
| `frontend`| nginx, на хосте по умолчанию **порт `FRONTEND_PORT`** (3000) |

Приложение в браузере: **`http://localhost:3000`** (или значение `FRONTEND_PORT`). Фронт проксирует `/api` и `/images` на бэкенд по внутреннему имени `backend:8080`.

Первый запуск с демо-данными (только на **пустой** БД):

```bash
# в .env
RUN_SEEDS=true
```

После успешного старта для продакшена верни `RUN_SEEDS=false`, чтобы при следующих деплоях не гонять сиды повторно.

## Переменные окружения

Файл **`.env`** в корне (шаблон — `.env.example`):

| Переменная | Назначение |
|------------|------------|
| `FRONTEND_PORT` | Порт nginx на хосте (по умолчанию 3000). |
| `DB_USER`, `DB_PASSWORD`, `DB_NAME` | Учётные данные Postgres; те же значения видит бэкенд. |
| `DB_PORT_HOST` | Проброс Postgres на хост (по умолчанию 5432). Меняй, если порт занят. |
| `JWT_SECRET` | **Обязательно смени в продакшене** — подпись access-токенов. |
| `JWT_ACCESS_TTL` | Время жизни JWT (например `168h`). |
| `RUN_SEEDS` | `true` — один раз применить `backend/seeds/dev.sql` при старте бэкенда. |

Секреты не коммить: `.env` в `.gitignore`.

## Данные и тома

| Том Docker | Содержимое |
|------------|------------|
| `lexis-db-data` | Данные Postgres — **бэкапить**, если важны пользователи и прогресс. |
| `lexis-images` | Загруженные картинки слов (`STORAGE_PATH` в контейнере бэкенда). |

---

## Продакшен: минимальный чеклист

1. Сильный **`JWT_SECRET`** и надёжный **`DB_PASSWORD`**.
2. **`RUN_SEEDS=false`**, если база уже живая.
3. Поставить **reverse proxy** (Caddy, nginx, Traefik) перед контейнером фронта: TLS, заголовки, при желании ограничение по IP.
4. Не публиковать **`DB_PORT_HOST`** наружу без необходимости — достаточно сети compose + при необходимости SSH-туннель для админки.

HTTPS обычно завершает прокси на хосте; менять `BACKEND_URL` во фронте в корневом compose **не нужно** для стандартной схемы «один хост, фронт шлёт относительные `/api`».

## Другие варианты Compose

| Файл | Назначение |
|------|------------|
| Корень: `docker-compose.yml` | **Рекомендуемый** полный стек. |
| `backend/docker-compose.yml` | Только **Postgres + backend** (без фронта), для API-only или отладки. |
| `frontend/docker-compose.yml` | Только **nginx со статикой**; бэкенд ожидается на хосте (`host.docker.internal:8080`). |

---

## Обновление после `git pull`

```bash
docker compose build --no-cache
docker compose up -d
```

При смене миграций бэкенд при старте прогоняет goose из образа; при проблемах смотри логи: `docker compose logs backend`.

---

## Локальная разработка без полного стека

- **Бэкенд:** из каталога `backend` — `go run ./cmd/server` при запущенном Postgres (см. `.env` в `backend/`).
- **Фронт:** из `frontend` — `npm run dev`, в `vite.config` обычно прокси на API.

Подробности по API и доменной модели — в основном `README.md`.
